Como funciona o teste?

Os testes de velocidade são realizados através de HTTP direto entre seu navegador e o servidor para medir precisamente a performance da sua conexão de rede.

Todos os testes são executados no cliente (seu navegador) medindo o tempo real de transferência entre sua máquina e o servidor.

Esta abordagem testa a velocidade real da sua conexão, não a velocidade do servidor ou loopback local.

Comando utilizado:
ping -c 1 meuip.ufabc.int.br
Processo:
  1. Executa 12 pings individuais (1 pacote cada)
  2. Intervalo de 500ms entre cada ping
  3. Extrai o tempo de resposta em milissegundos
  4. Apresenta cada resultado imediatamente
Resultado exemplo:

64 bytes from meuip.ufabc.int.br: icmp_seq=1 time=12.4 ms

O tempo medido é o round-trip time (RTT) - o tempo total da viagem de ida e volta.

Método utilizado:
fetch('/testfile') + performance.now()
Processo:
  1. Navegador baixa arquivo de teste de 1MB do servidor
  2. Mede tempo usando performance.now() (alta precisão)
  3. Calcula velocidade: (bytes × 8) / (tempo × 1.000.000)
  4. Testa velocidade real cliente-servidor
Exemplo de cálculo:

1.048.576 bytes em 0.125 segundos

Velocidade: (1.048.576 × 8) / (0.125 × 1.000.000)

Resultado: 67.1 Mbps

Método utilizado:
fetch('/upload', {method: 'POST', body: testData}) + performance.now()
Processo:
  1. JavaScript cria dados de teste de 1MB (Uint8Array)
  2. Navegador envia dados via POST para servidor
  3. Mede tempo usando performance.now() (alta precisão)
  4. Testa velocidade real cliente-servidor
Exemplo de cálculo:

1.048.576 bytes em 0.156 segundos

Velocidade: (1.048.576 × 8) / (0.156 × 1.000.000)

Resultado: 53.7 Mbps

Para especialistas em rede: Use os comandos abaixo para validar e comparar os resultados da interface web.
🔽 Teste de Download via Curl:
curl -s -o /dev/null -w "%{time_total},%{size_download},%{speed_download}" http://meuip.ufabc.int.br/testfile

Saída esperada: 0.125,1048576,67108864

Cálculo: 67.108.864 bytes/s = 67.1 Mbps

🔼 Teste de Upload via Curl:
dd if=/dev/zero bs=1024 count=1024 2>/dev/null | curl -s -X POST --data-binary @- -w "%{time_total},%{size_upload},%{speed_upload}" http://meuip.ufabc.int.br/upload

Saída esperada: 0.156,1048576,53687091

Cálculo: 53.687.091 bytes/s = 53.7 Mbps

🏓 Teste de Latência via Ping:
ping -c 12 meuip.ufabc.int.br

📊 Script Profissional de Validação:
Download do Script: validate_speed.sh
🚀 Como usar o script:
# Baixar, executar e remover automaticamente
curl -O http://meuip.ufabc.int.br/validate_speed.sh && chmod +x validate_speed.sh && ./validate_speed.sh

# Teste customizado (20 medições)
curl -O http://meuip.ufabc.int.br/validate_speed.sh && chmod +x validate_speed.sh && ./validate_speed.sh 20
Auto-Delete: O script se remove automaticamente após a execução para garantir que sempre seja baixada a versão mais atualizada.
📋 Funcionalidades do Script:
  • Múltiplas medições: Executa 12 testes por padrão (configurável)
  • Análise estatística: Calcula mediana (igual à interface web)
  • Retry automático: Tenta novamente em caso de falha
  • Cache-busting: Evita cache com timestamps únicos
  • Timeout: Evita travamento em conexões lentas
  • Auto-delete: Remove-se após execução (sempre versão atualizada)
  • Progress visual: Mostra progresso colorido

⚠️ Diferenças Esperadas:
🌐 INTERFACE WEB:
  • Método: HTTP fetch() via JavaScript
  • Timer: performance.now() (alta precisão)
  • Headers: Navegador + User-Agent
  • Conexão: HTTP/1.1 keep-alive
  • Retry: 3 tentativas automáticas
  • Estatística: Mediana de 12 testes
  • Cache: Cache-busting forçado
  • Overhead: Engine JavaScript + DOM
💻 CURL/SCRIPT:
  • Método: Curl nativo (C/libcurl)
  • Timer: Timer interno do curl
  • Headers: Mínimos (Host, User-Agent)
  • Conexão: HTTP/1.1 nova conexão
  • Timeout: 15s (download) / 30s (upload)
  • Estatística: Mediana de N testes
  • Cache: Cache-busting via timestamp
  • Overhead: Mínimo (nativo)
Diferenças Típicas:
  • ±5-15%: Variação normal entre métodos
  • Download: Curl pode ser 5-10% mais rápido (menos overhead)
  • Upload: Interface web pode ser mais rápida (keep-alive)
  • Latência: Ping sempre mais preciso que HTTP
  • Primeira execução: Curl pode ser mais lento (cold start)
  • Múltiplas execuções: Resultados devem convergir
Validação Profissional: Execute o script múltiplas vezes e compare as medianas. Diferenças >20% podem indicar problemas de rede ou configuração.
